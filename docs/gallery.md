# LLM-Wiki — What It Looks Like

Both humans and agents are first-class citizens of the same wiki. A human opens the vault in Obsidian and browses compiled knowledge through wikilinks. An agent connects via MCP and navigates the same files through budget-aware tools, reading inline maintenance signals and talk digests as part of every page read. No separate representation, no sync step.

This document shows what each artifact looks like from both sides.

---

## The Vault on Disk

```
~/wiki/
├── raw/                                          # Immutable source copies
│   ├── 2026-04-09-vaswani-attention-2017.pdf    # Original PDF (canonical ground truth)
│   └── 2026-04-09-vaswani-attention-2017.md     # Extracted text (agent food)
│
├── inbox/                                        # Active ingest plans (mutable, git-tracked)
│   └── 2026-04-09-vaswani-plan.md               # Deep ingest in progress
│
├── wiki/                                         # Compiled knowledge (daemon-owned)
│   ├── .issues/                                  # Persistent issue queue
│   │   └── broken-link-attention-mechanism-a1b2c3.md
│   ├── transformers/
│   │   ├── attention-mechanism.md
│   │   ├── attention-mechanism.talk.md           # Talk page sidecar (append-only)
│   │   ├── multi-head-attention.md
│   │   └── transformer.md
│   ├── protein-design/
│   │   ├── rfdiffusion.md
│   │   ├── bindcraft.md
│   │   └── bindsweeper.md
│   └── index.md                                  # Auto-generated by IngestAgent
│
└── schema/
    └── config.yaml                               # Backend profiles, routing, maintenance schedule
```

`raw/` is immutable source material. `inbox/` is mutable research-in-progress. `wiki/` is compiled knowledge the daemon owns. `schema/` is configuration. The daemon's own state (tantivy index, session journals, usage logs) lives in `~/.llm-wiki/` — never in the vault, never in git.

## A Wiki Page

### Human view — Obsidian / plain text

```markdown
---
title: Attention Mechanism
tags: [transformers, sequence-models, deep-learning]
summary: Scaled dot-product attention and multi-head variants; core operation of transformer architectures.
authority: 0.84
---

%% section: overview, tokens: 140 %%
## Overview

The [[attention mechanism]] allows a model to dynamically weight positions in a sequence
when producing a representation. Introduced by [[bahdanau-attention]] for
sequence-to-sequence translation, scaled dot-product attention was formalised in
[[attention-is-all-you-need]] as the core operation of the [[transformer]] architecture.

%% section: mechanism, tokens: 210 %%
## Mechanism

Given queries Q, keys K, and values V:

    Attention(Q, K, V) = softmax(QK^T / sqrt(d_k)) * V

The scaling factor `sqrt(d_k)` prevents dot products from growing large in
high-dimensional spaces. [[multi-head-attention]] runs H parallel heads over
projected subspaces and concatenates their outputs.

%% section: citations, tokens: 60 %%
## Citations

- [[raw/2026-04-09-vaswani-attention-2017.md]] §3.2 — original scaled dot-product formulation
- [[raw/2026-04-09-bahdanau-2015.md]] §3 — antecedent soft attention mechanism
```

Wikilinks (`[[...]]`) resolve to other pages in the vault. `%% section: name, tokens: N %%` markers are invisible in Obsidian preview — they give the daemon precise section boundaries for viewport-controlled reads. Without them, the daemon falls back to `##` heading-based slicing.

## Agent Traversal — What It Feels Like

The agent's reading pattern mirrors how a human researcher browses Wikipedia: orient first, then search, then read with purpose, then follow links that serve your intent. Token budgets replace screen pixels.

Here is a full traversal. The agent is trying to answer: *"How does RFDiffusion handle motif scaffolding, and does the wiki have any issues with its coverage?"*

---

**Step 1 — Orient: `wiki_manifest(budget=800)`**

```json
{
  "status": "ok",
  "content": "protein-design (6 pages, 18400 tokens)\n  RFDiffusion, BindCraft, ProteinMPNN, BindSweeper, evaluation methods\ntransformers (3 pages, 8420 tokens)\n  Core attention and transformer architecture\nmachine-learning (5 pages, 11200 tokens)\n  Optimisation, evaluation metrics, clustering"
}
```

The agent sees the cluster landscape without loading any page content. `protein-design` is the target. Budget consumed: ~80 tokens.

---

**Step 2 — Search: `wiki_search("motif scaffolding", limit=3)`**

```json
{
  "status": "ok",
  "results": [
    {
      "name": "rfdiffusion",
      "score": 0.94,
      "manifest": "rfdiffusion: Diffusion-based protein structure generation with multiple design modes including de novo, partial diffusion, and motif scaffolding.",
      "matches": [
        {"line": 42, "before": "For binder design,", "match": "motif scaffolding", "after": "constrains a fixed structural motif"}
      ]
    },
    {
      "name": "bindsweeper",
      "score": 0.31,
      "manifest": "bindsweeper: Multi-dimensional parameter sweep tool for protein design pipeline optimisation.",
      "matches": []

  ]
}
```

Clear hit on `rfdiffusion`. Agent skips `bindsweeper` — low score, no match.

The agent sees a critical issue immediately. It cannot answer the research question without acknowledging this. The maintenance backlog is a research finding.

**Step 4 — Read the relevant section: `wiki_read("rfdiffusion", viewport="section", section="motif-scaffolding")`**


The agent can now answer the question and notes for follow-up: the critical adversary finding on `rfdiffusion` (success rate claim contradicted) should be escalated to the user or resolved with a `wiki_talk_post` before any new content is written to that page.

## The Manifest at Different Budgets

The manifest is the agent's map of the vault. Budget controls depth: low budget gives cluster summaries, higher budget unfolds page entries and section detail. The agent starts shallow and only goes deeper where it needs to.

### Budget 500 — cluster summaries only

```
protein-design (6 pages, 18400 tokens)
  RFDiffusion, BindCraft, ProteinMPNN, FamPNN, BindSweeper, Boltz-2 evaluation
transformers (3 pages, 8420 tokens)
  Core attention mechanism, multi-head attention, transformer architecture
machine-learning (5 pages, 11200 tokens)
  Optimisation methods, evaluation metrics, clustering approaches
```

The agent sees the shape of the vault in ~80 tokens. Enough to know which cluster to enter.

### Budget 2000 — page entries unfolded

```
protein-design (6 pages, 18400 tokens)
  rfdiffusion: Diffusion-based protein structure generation; de novo, partial diffusion, motif scaffolding
    tags: [protein-design, diffusion, structure-generation] | tokens: 4200 | authority: 0.88
    sections: [overview(320t), design-modes(890t), motif-scaffolding(1100t), contigs-syntax(640t)]
    links: [bindcraft, contigs-syntax, rfdiffusion2]
  bindcraft: Hallucination-based binder design via backpropagation through AlphaFold2
    tags: [protein-design, hallucination, binder-design] | tokens: 3100 | authority: 0.76
    sections: [overview(280t), induced-fit(740t), protocols(810t)]
    links: [rfdiffusion, alpha-fold-2, protein-mpnn]
  bindsweeper: Multi-dimensional parameter sweep framework for protein design pipelines
    tags: [protein-design, hyperparameter-optimisation, tooling] | tokens: 2400 | authority: 0.61
    sections: [overview(190t), yaml-config(580t), sweep-matrix(830t)]
    links: [rfdiffusion, bindcraft, protein-mpnn]
```

Now the agent can see which pages exist, what they cover, how large they are, and how they connect — without reading any page body.

The `authority` score is computed by the librarian from the link graph and usage patterns. High-authority pages are the knowledge hubs; the agent and the adversary both weight them accordingly.

## A Talk Page

Talk pages are append-only sidecars: one per wiki page, stored alongside it as `<page>.talk.md`. They accept anything — adversary findings, librarian suggestions, human notes, agent proposals — with no citation requirement. Main pages are sourced; talk pages are everything pre-source.

### Human view — `wiki/transformers/attention-mechanism.talk.md`

```markdown
---
page: attention-mechanism
---

**2026-04-07T15:22:10.456789+00:00 — @librarian** <!-- severity:suggestion -->
Consider cross-linking [[positional-encoding]] from the Mechanism section —
frequently co-traversed in usage logs (8 joint reads this week).

**2026-04-08T11:45:33.123456+00:00 — @adversary** <!-- severity:moderate -->
Claim verified: "scaling factor prevents dot products from growing large in
high-dimensional spaces" — checked against raw/2026-04-09-vaswani-attention-2017.md §3.2.1.
Verdict: SUPPORTED. Last corroborated: 2026-04-08.

**2026-04-09T09:11:20.654321+00:00 — @researcher** <!-- resolves:[1] -->
Added cross-link to [[positional-encoding]] in Mechanism section. Resolves librarian suggestion.
```

Entry 1 is now closed — entry 3 resolves it via `resolves:[1]`. The original entry stays as audit trail. The open set is just entry 2 (the adversary finding, which is informational — SUPPORTED means no action needed).

### Agent view — inline digest in `wiki_read`

The agent never reads the `.talk.md` file directly unless it calls `wiki_talk_read`. Instead, a digest appears in every `wiki_read` response:

```json
"talk": {
  "cnt": 3,
  "open": 1,
  "sev": {"critical": 0, "moderate": 1, "suggestion": 0},
  "summary": "Adversary verified scaling factor claim (SUPPORTED, 2026-04-08). Librarian cross-link suggestion resolved.",
  "crit": [],
  "mod": [
    {
      "index": 2,
      "author": "@adversary",
      "body": "Claim SUPPORTED: scaling factor rationale verified against Vaswani §3.2.1."
    }
  ]
}
```

The moderate adversary finding appears verbatim. The resolved suggestion collapses into the librarian summary. The agent sees the maintenance state of the page without a separate tool call.

## A Raw Source

Sources are copied verbatim into `raw/` before ingestion — the original file is the canonical ground truth. The frontmatter is mutable metadata; the body is immutable source content.

```markdown
---
reading_status: in_progress
ingested: 2026-04-09
source_type: paper
title: "Attention Is All You Need"
authors: [Vaswani, Shazeer, Parmar, Uszkoreit, Jones, Gomez, Kaiser, Polosukhin]
year: 2017
---

[Extracted text — verbatim from pdf_extractor]

Abstract

We propose a new simple network architecture, the Transformer, based solely on
attention mechanisms, dispensing with recurrence and convolutions entirely.
Experiments on two machine translation tasks show these models to be superior
in quality while being more parallelizable and requiring significantly less
time to train...
```

`reading_status` starts as `unread` on copy. The daemon sets this; the researcher never edits the header manually. Attended engagement — Brief or Deep mode — promotes it to `in_progress` then `read` via `wiki_source_mark`. Autonomous ingest can never promote a source past `unread`.

## An Inbox Plan File

The inbox plan file is the persistent cursor for a Deep ingest spanning multiple sessions. Created before any wiki write, committed to git after each session. The next session reads it and reconstructs the task list — no prior context needed.

```markdown
---
source: raw/2026-04-09-vaswani-attention-2017.pdf
started: 2026-04-09
status: in-progress
sessions: 2
---

# Attention Is All You Need — Research Plan

## Claims / Ideas
- [x] Scaled dot-product attention — core mechanism, d_k scaling rationale
- [x] Multi-head attention — parallel heads, projection subspaces, concatenation
- [ ] Positional encoding — sine/cosine basis, learned vs fixed tradeoffs   ← resume here
- [ ] Encoder-decoder architecture — layer structure, residuals, layer norm
- [ ] Training regime — Adam with warmup, label smoothing, dropout schedule

## Decisions
- attention-mechanism: new page ✓ (no existing coverage)
- multi-head-attention: new page ✓ (distinct from attention-mechanism)
- transformer: new page ✓ (central to source)
- training-regime: check optimisation cluster first before creating

## Session Notes

### 2026-04-09 (Session 1)
Covered scaled dot-product attention and multi-head attention. Found no existing
coverage — clean slate. Created [[attention-mechanism]], [[multi-head-attention]].
Noted: [[bahdanau-attention]] referenced but no page exists — filed as moderate issue.

### 2026-04-10 (Session 2)
Resuming at positional encoding. User flagged the sine/cosine derivation should
be captured precisely, not just linked out. Will write a full section.
```

The task list tracks *claims and ideas*, not pages. The page is an output of the analysis, not the unit of work. Some claims become pages, some become talk posts, some stay as decisions in the plan file.

## Brief Mode Output

Brief mode is what happens when the attended agent reads a source with the researcher's full context loaded — their wiki, their memory, their prior work. The output is a briefing, not pages. The researcher decides what to do next.

```
**New to your work:** The paper's §3.2.1 explanation of why d_k scaling is necessary
(gradient flow argument in high-dimensional dot products) is more rigorous than your
existing [[attention-mechanism]] note — worth updating. Positional encoding and the
training regime (warmup schedule, label smoothing) have no wiki coverage at all.

**Already covered:** Scaled dot-product attention ([[attention-mechanism]]), encoder-decoder
architecture ([[transformer]]). This paper is the primary source for both — your
existing pages cite it correctly.

**Contradictions:** None found against existing pages.

**Worth reading yourself:** §6 (training details, p.8-9) — the warmup schedule
rationale involves empirical tradeoffs you'll want to evaluate before writing the
training-regime page. The paper doesn't fully justify the specific hyperparameters.

**Scope if queued:** ~3 pages (positional-encoding, training-regime, layer-norm)
```

The briefing is the value. Page creation is optional and researcher-directed. The agent has read a 15-page paper, checked the vault for coverage, and distilled it into five lines that tell the researcher exactly what they need to know.

## A Session Git Commit

Every supervised write session settles into a single git commit. The daemon's cheap LLM writes the subject and bullets from the session journal; if the LLM is unreachable, a deterministic fallback message is used — **the commit always happens**.

```
commit 3e7f9a1b
Author: llm-wiki daemon <daemon@local>
Date:   Thu Apr 10 14:23:11 2026 +0100

    wiki: Document transformer attention mechanism and multi-head variants

    - page-create wiki/transformers/attention-mechanism.md — Core scaled dot-product formulation, d_k scaling rationale
    - page-create wiki/transformers/multi-head-attention.md — Parallel attention heads, projection subspaces, concatenation output
    - page-append wiki/transformers/transformer.md — Added cross-references to new attention pages

    Session: 3e7f9a1b-4c2d-4e8f-9a1b-3e7f9a1b4c2d
    Agent: claude-researcher
    Writes: 7
```

The `Agent:` trailer makes the git log a meaningful audit trail. `git log --grep "Agent: claude-researcher"` returns every session that agent has run. For a swarm, `git log --grep "Agent: worker-"` returns all worker output. No shadow log, no provenance frontmatter — the commit history is the record.

## The MCP Tool Surface

17 tools across four families. Register `llm-wiki mcp` as a stdio MCP server and these become available in any MCP-capable agent framework.

### Read

| Tool | What it does |
|------|-------------|
| `wiki_search` | Keyword search across the vault. Returns ranked results with match context and manifest entry per page. Limit 1–50 (default 10). |
| `wiki_read` | Read a page with viewport control: `top` (first section + TOC), `section` (named section), `grep` (pattern match), `full` (entire page). Always includes inline issue and talk digests. |
| `wiki_manifest` | Budget-aware hierarchical index of the vault. Low budget → cluster summaries. Higher budget → page entries with tags, authority, sections, links. |
| `wiki_status` | Vault stats: page count, cluster count, total tokens, last index time. |

### Query

| Tool | What it does |
|------|-------------|
| `wiki_query` | Ask a question. The daemon runs multi-turn traversal internally and returns a synthesised answer with citations. Near-zero context cost to the calling agent. |
| `wiki_ingest` | Ingest a source (PDF, DOCX, markdown, image). Supports `dry_run=true` to preview concept extraction without writing. |
| `wiki_lint` | Structural integrity check: orphans, broken wikilinks, broken citations, missing section markers. Returns vault-wide attention map. |

### Write

| Tool | What it does |
|------|-------------|
| `wiki_create` | Create a new page. Citations required. Rejects near-matches by default (`force=true` to override after confirming the page is genuinely distinct). |
| `wiki_update` | Apply a V4A patch to an existing page. Exact match with fuzzy fallback for minor drift. Returns structured `patch-conflict` on failure — re-read and retry, never rewrite the whole page. |
| `wiki_append` | Append a section to an existing page, anchored to a heading. Citations required. Safest write option. |

### Maintenance

| Tool | What it does |
|------|-------------|
| `wiki_issues_list` | List open issues. Filter by status (`open`/`resolved`/`wontfix`) and type (`broken-link`, `broken-citation`, `orphan`, `claim-failed`, `compliance`, `missing-markers`). |
| `wiki_issues_get` | Read full issue body by ID. |
| `wiki_issues_resolve` | Mark an issue resolved or wontfix. |
| `wiki_talk_read` | Read the full talk page thread for a wiki page. |
| `wiki_talk_post` | Post a talk entry. Severities: `critical`, `moderate`, `minor`, `suggestion`, `new_connection`. |
| `wiki_talk_list` | List pages that have talk entries, with open counts by severity. |
| `wiki_session_close` | Explicitly settle the current write session into a git commit. Also happens automatically on inactivity timeout (5 min), write cap (30 writes), or daemon shutdown. |

Every write tool requires an `author` identifier. Writes without one are rejected. All writes in a session settle into a single git commit with `Session:`, `Agent:`, and `Writes:` trailers.
